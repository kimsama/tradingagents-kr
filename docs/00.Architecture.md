# TradingAgents — Architecture

This document describes how the TradingAgents framework is organised: the high-level pipeline, the runtime state machine, and the cross-cutting modules (LLM clients, data flows, memory log, checkpoint store) that the agents depend on. It is meant for contributors who need to understand or change the system, not for end users — see `docs/10.USER_MANUAL_KO.md` for operational guidance.

## 1. System overview

TradingAgents is a **multi-agent stock-analysis pipeline** built on top of [LangGraph](https://github.com/langchain-ai/langgraph). A run is parameterised by `(ticker, trade_date)` and produces:

- per-section markdown reports (market, sentiment, news, fundamentals, research, trading, risk, portfolio)
- a structured final trade decision (`Buy / Overweight / Hold / Underweight / Sell`)
- a persisted memory-log entry that subsequent runs read for context

The framework is provider- and vendor-agnostic: any supported LLM provider (OpenAI, Anthropic, Google, xAI, DeepSeek, Qwen, GLM, OpenRouter, Azure, Ollama) can drive any agent, and any supported data vendor (yfinance, Alpha Vantage) can serve the underlying market data.

```
                    ┌────────────────────────────────────────┐
                    │              CLI (cli/main.py)         │
                    │  questionary prompts → config dict     │
                    └────────────────────┬───────────────────┘
                                         │ TradingAgentsGraph(config)
                                         ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                       TradingAgentsGraph (orchestrator)                  │
│                                                                          │
│  • Resolves OAuth/API-key auth, builds quick + deep LLM clients          │
│  • Wires tool nodes for each analyst category                            │
│  • Compiles the LangGraph workflow (with optional SqliteSaver)           │
│  • Resolves pending memory-log entries before running                    │
│  • Streams or invokes the graph, then logs state + stores decision       │
└────────────────────┬─────────────────────────────────────────────────────┘
                     │ propagate(ticker, trade_date)
                     ▼
            ┌────────────────────────────┐
            │   LangGraph workflow       │
            │   StateGraph(AgentState)   │
            └─────────────┬──────────────┘
                          │
                          ▼
        Analysts → Researchers → Trader → Risk → Portfolio Manager
```

## 3. The pipeline (LangGraph state machine)

The agents are wired as nodes on a single `StateGraph(AgentState)`. The shape of the graph is built in `tradingagents/graph/setup.py`; routing decisions are made by methods on `ConditionalLogic` (`tradingagents/graph/conditional_logic.py`).

```
START ──► [Market Analyst] ─► [Social] ─► [News] ─► [Fundamentals]
            │      ▲              (each analyst loops with its tool node
            ▼      │               via add_conditional_edges until no
         [tools_*] │               more tool calls are emitted)
            │      │
            └──────┘
                                ▼
                        [Bull Researcher] ◄─────────────┐
                                │                       │
                                ▼ (count % 2)           │
                        [Bear Researcher] ──────────────┘
                                │
                                ▼ (count >= 2 * max_debate_rounds)
                        [Research Manager]
                                │
                                ▼
                            [Trader]
                                │
                                ▼
                       [Aggressive Analyst] ◄─────────┐
                                │                     │
                                ▼ (latest_speaker)    │
                       [Conservative Analyst] ────────┤
                                │                     │
                                ▼                     │
                       [Neutral Analyst] ─────────────┘
                                │
                                ▼ (count >= 3 * max_risk_discuss_rounds)
                       [Portfolio Manager]
                                │
                                ▼
                              END
```

Important properties:

- **Selected analysts are configurable** (`selected_analysts` arg), but the pipeline always continues through Research → Trader → Risk → Portfolio. Picking only "market" still produces a final trade decision.
- **Tool-using analysts loop**: each analyst node returns either a tool call (routed to its `tools_*` `ToolNode`) or a final report (routed through a `Msg Clear *` node that strips the working messages before handing off to the next analyst). This keeps each analyst's working memory bounded.
- **Debate routing uses parity, not display text**. `should_continue_debate` decides Bull/Bear from `count % 2`, not from prefixes on `current_response`. This was changed in PR #5 so that localised speaker labels (e.g. `강세 분석가`) cannot break control flow.
- **Risk routing uses `latest_speaker` prefix** because the three-way rotation is harder to express as a parity check; the speaker tags are stable English schema strings (`"Aggressive ..."`, `"Conservative ..."`).

## 4. Shared state — `AgentState`

Defined in `tradingagents/agents/utils/agent_states.py`. Inherits from LangGraph's `MessagesState`, so it carries the in-flight message list plus typed report fields:

| Field | Producer | Notes |
|---|---|---|
| `company_of_interest`, `trade_date` | initial state | passthrough identifiers |
| `market_report`, `sentiment_report`, `news_report`, `fundamentals_report` | analyst nodes | free-text markdown |
| `investment_debate_state` (`InvestDebateState`) | bull/bear/research-manager | `bull_history`, `bear_history`, `current_response`, `count`, `judge_decision` |
| `investment_plan` | research manager | rendered from `ResearchPlan` schema |
| `trader_investment_plan` | trader | rendered from `TraderProposal` schema |
| `risk_debate_state` (`RiskDebateState`) | risk-mgmt nodes | per-perspective histories + `latest_speaker` + `count` |
| `final_trade_decision` | portfolio manager | rendered from `PortfolioRating` schema |
| `past_context` | injected by orchestrator | memory-log excerpts for same-ticker + cross-ticker lessons |

Schema labels in this state are **always English**, even when the user selects Korean output. The English schema is what parsers, the memory log, and routing all consume. Korean rendering happens later, in `cli/main.py`'s `localize_report_markdown` post-processing pass.

## 5. Two-layer LLM strategy: prose vs structured output

`tradingagents/agents/schemas.py` documents the design choice: **most agents emit prose**, but the three decision-making agents (Research Manager, Trader, Portfolio Manager) use **provider-native structured output** via Pydantic schemas:

| Agent | Schema | Render helper |
|---|---|---|
| Research Manager | `ResearchPlan` (recommendation, rationale, strategic actions) | `render_research_plan` |
| Trader | `TraderProposal` (action, reasoning, entry/stop/sizing) | `render_trader_proposal` |
| Portfolio Manager | structured rating + thesis + targets | per-schema renderer |

`invoke_structured_or_freetext` (in `agents/utils/structured.py`) wraps the structured call so that:

1. If structured output succeeds, the Pydantic instance is rendered to markdown.
2. If it fails *not* due to a rate limit, the system falls back to a free-text invocation with the same prompt.
3. **If it fails due to rate-limit**, the fallback is *not* attempted — re-issuing the same prompt would only worsen quota pressure.

The render helpers always produce the canonical English schema markdown (`**Recommendation**: ...`, `**Action**: ...`). Localisation happens at display/save time, not at agent boundaries.

## 6. Provider abstraction — `tradingagents/llm_clients/`

`create_llm_client(provider, model, base_url=None, *, auth_mode=None, **kwargs)` is the single entry point. It:

1. Resolves `auth_mode` precedence: explicit arg → `<PROVIDER>_AUTH_MODE` env var → `"api_key"`.
2. Rejects OAuth for providers that don't support it (only `anthropic` and `openai` are OAuth-capable today).
3. **Lazily imports** the per-provider client module — simply importing the factory does not pull in heavy SDKs or fail when API keys are missing. This keeps test collection fast.

| Provider family | Client module | Underlying SDK |
|---|---|---|
| OpenAI + OpenAI-compatible (`xai`, `deepseek`, `qwen`, `glm`, `openrouter`, `ollama`) | `openai_client.py` | `langchain-openai` (`ChatOpenAI`) |
| Anthropic | `anthropic_client.py` | `langchain-anthropic` (`ChatAnthropic`) |
| Google Gemini | `google_client.py` | `langchain-google-genai` |
| Azure OpenAI | `azure_client.py` | `langchain-openai` (Azure variant) |

### 6.1 OAuth (subscription) mode

For Anthropic and OpenAI, users can run against their Claude Pro/Max or ChatGPT subscription instead of paying per-token. The OAuth adapters (`oauth/anthropic_oauth.py`, `oauth/openai_oauth.py`) are **read-only**: they parse the credential file written by the source CLI (`~/.claude/.credentials.json`, `~/.codex/auth.json`) and expose an `httpx.Client` / `httpx.AsyncClient` preconfigured with the right bearer token, beta headers, and `x-api-key` strip.

Refresh is intentionally delegated to the source CLI (`claude` / `codex login`); the adapter raises an actionable `*OAuthError` if the cached token is expired.

The Anthropic side requires a load-bearing `_client` / `_async_client` override on `NormalizedChatAnthropic` because `langchain_anthropic.ChatAnthropic` always rebuilds its own `httpx.Client` and discards anything passed via `model_kwargs`. Without the override, OAuth headers would be silently dropped.

### 6.2 Rate-limit handling

`tradingagents/rate_limit.py` is the single 429-detection helper, used by:

- `NormalizedChatAnthropic.invoke` / `ainvoke` — wraps the call in a 30s/90s retry loop (configurable via `ANTHROPIC_RATE_LIMIT_RETRY_SECONDS`), honouring the `Retry-After` header when present.
- `agents/utils/structured.py` — disables free-text fallback on 429 to avoid worsening quota pressure.

`is_rate_limit_error` checks status code, exception class name, and a text-substring fallback (in that order of preference). The async retry uses `await asyncio.sleep(...)` so the event loop is not blocked.

## 7. Data layer — `tradingagents/dataflows/`

Agents call abstract tool functions defined in `agents/utils/*_tools.py`. Each tool dispatches to a vendor-specific implementation in `dataflows/` using two layers of routing in `default_config.py`:

```python
"data_vendors": {
    "core_stock_apis":      "yfinance",       # category-level default
    "technical_indicators": "yfinance",
    "fundamental_data":     "yfinance",
    "news_data":            "yfinance",
},
"tool_vendors": {
    # "get_stock_data": "alpha_vantage",      # per-tool override (takes precedence)
},
```

Supported vendors today: `yfinance` (free, default) and `alpha_vantage` (requires `ALPHA_VANTAGE_API_KEY`). Adding a vendor means adding a `dataflows/<vendor>*.py` module, registering its functions in `dataflows/__init__.py`, and updating the dispatcher in `interface.py`.

## 8. Persistence

Three persistence concerns are kept separate:

| Store | Purpose | Path | Format |
|---|---|---|---|
| **Saved reports** | User-facing markdown output of each run | `${results_dir}/<TICKER>/<DATE>/reports/*.md` + `complete_report.md` | Markdown (localised at save time) |
| **State log** | Full final state of each run | `${results_dir}/<TICKER>/TradingAgentsStrategy_logs/full_states_log_<DATE>.json` | JSON |
| **Memory log** | Append-only ledger of decisions + post-trade reflections | `${memory_log_path}` (default `~/.tradingagents/memory/trading_memory.md`) | Markdown with `<!-- ENTRY_END -->` separators |
| **Checkpoint** | LangGraph resume state (opt-in via `checkpoint_enabled`) | `${data_cache_dir}/checkpoints/<TICKER>.db` | SQLite (per-ticker) |

### 8.1 Memory log lifecycle

`TradingMemoryLog` (`agents/utils/memory.py`) is intentionally simple — append-only markdown, no LLM call on the write path:

1. **Write (end of run)**: `store_decision(...)` appends `[date | ticker | rating | pending]` followed by `DECISION:\n<markdown>`.
2. **Resolve (start of next same-ticker run)**: `_resolve_pending_entries(ticker)` fetches realised return + alpha vs SPY, generates a reflection via `Reflector.reflect_on_final_decision`, and rewrites pending entries to resolved.
3. **Read (initial state)**: `get_past_context(ticker)` returns formatted excerpts (same-ticker recent + cross-ticker lessons), injected as `past_context` into `AgentState` for the Portfolio Manager.

Entries that can't be resolved (price not yet available, delisted, network error) stay pending and are retried on subsequent runs.

### 8.2 Checkpoints

Opt-in (`checkpoint_enabled: true`). When enabled, `TradingAgentsGraph.propagate` recompiles the workflow with a `SqliteSaver` backed by a per-ticker DB. The thread ID is `sha256("TICKER:DATE")[:16]`, so the same ticker+date resumes while a new date starts fresh. On successful completion the checkpoint is cleared to avoid stale state.

Per-ticker DBs (rather than one global DB) avoid SQLite write contention when running multiple tickers concurrently.

## 9. Localisation

Two-layer design:

1. **Prompt-level instruction** (`get_language_instruction()` in `agents/utils/agent_utils.py`) — appended to user-facing agent prompts so the LLM writes its prose in the selected language.
2. **Post-processing pass** (`localize_report_markdown(text, lang)`) — deterministic string/regex replacement of schema labels (`**Recommendation**` → `**추천 의견**`, `FINAL TRANSACTION PROPOSAL: **BUY**` → `최종 거래 제안: **매수**`, etc.) applied at display/save time only.

This is deliberate: the agent state and memory log keep canonical English labels, so parsers and downstream agents are not affected by the user's language choice. Adding a new language is a registry addition (`_MARKDOWN_LOCALIZATIONS["<lang>"] = {...}`), not a code change in every agent.

## 10. CLI flow

`cli/main.py` is a Typer app. The interactive `analyze` command:

1. Walks the user through 8 questionary prompts (ticker → date → language → analysts → research depth → provider → quick/deep models → provider-specific config).
2. Builds a `config` dict on top of `DEFAULT_CONFIG`.
3. Constructs `TradingAgentsGraph(selected_analysts, config, callbacks=stats_handler.callbacks)`.
4. Calls `propagate(ticker, date)` and streams progress through `MessageBuffer`, which feeds the live Rich console panels (Progress, Messages & Tools, Current Report, Footer counters).
5. On completion, prompts to save the consolidated report to `${results_dir}/manual_reports/<TICKER>_<TIMESTAMP>/`.

## 11. Configuration precedence

For any setting that has multiple sources, the precedence is:

```
explicit kwarg in code  >  config dict passed to TradingAgentsGraph
                        >  environment variable
                        >  DEFAULT_CONFIG value
                        >  hard-coded default in the consuming module
```

Concrete examples:

- `auth_mode`: `create_llm_client(auth_mode="oauth")` > `ANTHROPIC_AUTH_MODE` env var > `"api_key"` default.
- `results_dir`: `config["results_dir"]` > `TRADINGAGENTS_RESULTS_DIR` env var > `~/.tradingagents/logs`.
- Anthropic retry delays: `ANTHROPIC_RATE_LIMIT_RETRY_SECONDS="30,90"` > built-in `(30.0, 90.0)`.

## 12. Extensibility cheat-sheet

| Adding a … | Touch points |
|---|---|
| **New LLM provider** | Add `llm_clients/<provider>_client.py` subclassing `BaseLLMClient`; register in `factory.py`; add to `model_catalog.py` for CLI picker; add validator in `validators.py`. |
| **New data vendor** | Add `dataflows/<vendor>*.py`; register functions in `dataflows/__init__.py`; extend dispatcher in `dataflows/interface.py`; add option to `default_config.py`'s `data_vendors`. |
| **New agent** | Add `agents/<group>/<agent>.py` exposing a `create_<agent>(llm)` factory; export from `agents/__init__.py`; add a node + edges in `graph/setup.py`; add a routing function in `graph/conditional_logic.py` if it's a branch point. |
| **New output language** | Add an entry to `_MARKDOWN_LOCALIZATIONS` in `agents/utils/agent_utils.py`; extend `is_<lang>_output_language` (or refactor to a per-language predicate); add the language choice to `cli/utils.py:ask_output_language`. |
| **New structured-output schema** | Add Pydantic model + render helper to `agents/schemas.py`; bind in the agent factory via `bind_structured`; gate fallback behaviour with `invoke_structured_or_freetext`. |

## 14. Deployment posture

- **CLI / single-tenant**: the design assumption today. Per-`get_llm()` clients, blocking 30s/90s retries, and `time.sleep`-based backoff are acceptable for a desktop or container CLI.
- **Long-lived service**: not yet supported. OAuth `httpx.Client` instances are created per `get_llm()` call and never explicitly closed; the retry loop blocks the event loop on the sync path. Both are tracked as follow-up items in the OAuth PR thread.

For Docker layout (named volume + entrypoint priv-drop), see `docker-entrypoint.sh` and `docs/10.USER_MANUAL_KO.md`.
